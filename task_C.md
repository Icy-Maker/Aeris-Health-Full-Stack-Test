# 任务C：Xero API 场景问题解答
参考官方文档：[Xero Accounting API 官方总览](https://developer.xero.com/documentation/api/accounting/overview)

---

### C1. 在查询发票数据之前，如何验证 Xero API 的连接是否正常？
在执行业务数据查询前，优先通过轻量接口完成连通性与鉴权校验，步骤如下：
1. **优先调用连接状态接口**
   携带合法的授权请求头，调用 `GET /connections` 接口。
   - 若返回 200 状态码及已连接的租户（组织）列表，说明访问凭证有效、鉴权逻辑正常、API 服务可正常访问。
   - 该接口无额外业务权限要求，是专门用于验证连通性的最轻量方案。

2. **补充验证：调用基础资源接口**
   携带正确的 `Xero-Tenant-Id` 请求头，调用 `GET /Organisation` 组织信息接口。
   - 若成功返回 200 状态码，可进一步确认目标租户可访问、凭证具备基础读取权限，完成端到端的可用性验证。

---

### C2. 如果 /connections 接口调用正常，但 GET /Invoices 接口调用失败，应排查哪些内容？
按照「权限→参数→请求→业务」的层级逐步排查：
1. **授权范围校验**
   检查访问令牌的 OAuth 权限范围是否包含发票读取权限（如 `accounting.transactions.read`）。`/connections` 仅需基础鉴权即可通过，而发票接口需要特定的业务数据权限。

2. **租户请求头校验**
   确认请求头中的 `Xero-Tenant-Id` 填写正确，且与 `/connections` 返回的有效租户ID一致。缺失、填错或租户已失效，都会导致发票请求失败，但不影响连接接口的调用。

3. **请求参数与格式校验**
   核查所有查询参数（过滤条件、分页参数、日期范围等）：
   - 过滤语句语法错误、日期格式不合法、分页数值超出范围，都会导致请求被拒绝；
   - 同时确认请求头（如 `Accept`、`Content-Type`）符合 Xero API 的版本规范。

4. **用户权限校验**
   确认授权账号在目标组织内具备发票数据的查看权限，部分用户角色对财务数据有访问限制。

5. **限流状态校验**
   检查响应是否返回 429 状态码，确认是否因请求频率超出 Xero API 限流阈值而被拒绝。

6. **数据与过滤条件校验**
   确认目标组织内存在发票数据，且查询过滤条件未将全部数据排除。

### C3. 查询发票列表需要调用哪个接口端点？
查询发票列表，调用 Xero 会计API下的 **`GET /Invoices`** 端点。
- 完整基础路径：`https://api.xero.com/api.xro/2.0/Invoices`
- 该接口支持分页、状态筛选、日期范围筛选等查询能力，可批量获取发票记录。

---

### C4. 如何查询某一张特定的发票？
查询单张指定发票，需将发票的唯一标识 `InvoiceID`（发票的全局唯一GUID）作为路径参数，传入发票详情接口：
- 接口格式：**`GET /Invoices/{InvoiceID}`**
- 将 `{InvoiceID}` 替换为目标发票的实际GUID即可。
- 该接口会返回单张发票的完整详情，包含明细行、状态、付款记录、附件等全部关联信息。

---

### C5. 如果发票接口返回 429 状态码，后端应当如何处理？
HTTP 429 代表请求超出 Xero API 的限流阈值，后端应按以下规则处理：
1. **遵循 Retry-After 头重试**
   优先读取响应头中的 `Retry-After` 字段，在指定时长内暂停后续所有 Xero API 请求，到期后再重试。这是最合规、最高效的处理方式。

2. **带抖动的指数退避重试**
   若响应无 `Retry-After` 字段，采用**带随机抖动的指数退避算法**进行重试：初始等待较短间隔，连续失败时等待时长翻倍，避免持续打满限流阈值。

3. **后端主动限流管控**
   在后端侧对 Xero API 调用实现本地限流器与请求队列，平滑突发流量，从源头主动避免触发限流，而非仅被动处理报错。

4. **业务降级与优雅失败**
   - 非实时任务（如数据同步）：放入异步任务队列，后续自动重试，不阻塞主流程；
   - 实时请求：向前端返回友好的业务提示，而非抛出原始接口错误，告知用户稍后重试。

5. **日志与监控**
   记录 429 事件的时间、请求详情，纳入监控体系，用于后续优化请求频率与业务逻辑。
